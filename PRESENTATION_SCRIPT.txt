SPELL CHECKER: 3-STAGE ALGORITHM PIPELINE
================================================================================

================================================================================
OPENING (0:00 - 0:30)
================================================================================

"Hello everyone. I'm presenting a spell-checker system—a practical application 
of algorithmic design.

The core challenge: **Users type words incorrectly. How do we suggest the right 
correction efficiently?**

For example, 'recieve' should become 'receive'. The naive approach—comparing 
against all 554 dictionary words—is too slow. Our solution: a 3-stage pipeline 
that progressively filters candidates."

================================================================================
STAGE A: FAST LOOKUP (0:30 - 1:15)
================================================================================

"**Stage A: Hash Set Dictionary Lookup**

We store all valid words in a Python hash set for O(1) average-case lookup.

If 'receive' is typed, we find it immediately and return 'correct: true'.
If 'recieve' is typed, we don't find it and move to Stage B.

Why a hash set over a binary tree? Hash sets give us O(1) lookup for our sole 
operation—membership testing. A BST would be O(log n). The hash set is faster."

================================================================================
STAGE B: EDIT DISTANCE (1:15 - 3:30)
================================================================================

"**Stage B: Levenshtein Distance & Candidate Generation**

This is where most of the work happens. Two steps:

Step 1—Generate candidates:
We create all strings exactly one edit operation away—deletions, insertions, 
replacements, transpositions.

For 'recieve' (7 letters), that's roughly 7 + (7 × 26) + (7 × 26) + 7 = 
about 400 candidates. Time: O(L² × 26) where L is word length.

Step 2—Compute actual distance:
We use dynamic programming to calculate the Levenshtein distance between the 
input and each dictionary candidate.

The idea: Build a matrix where entry (i,j) is the edit distance between the 
first i characters of word A and the first j characters of word B.

The recurrence:
  - If characters match: dp[i][j] = dp[i-1][j-1]
  - Else: dp[i][j] = 1 + min(delete, insert, replace)

For two 7-letter words, that's 49 operations per pair.

Why dynamic programming? Without it, we'd recompute the same subproblems 
exponentially. With DP, each cell is computed once.

**Key insight:** We don't compare against ALL 554 words. The candidate generation 
stage filters us down to ~50 plausible matches. Then we only run the expensive 
DP algorithm on those 50, not all 554.

For 'recieve', the transposition step generates 'receive'—which IS in the 
dictionary. So we get the right answer."

================================================================================
STAGE C: RANKING (3:30 - 4:30)
================================================================================

"**Stage C: Heap-Based Top-K Ranking**

We have ~50 candidate words with their edit distances. Now rank them.

We use a binary min-heap via Python's heapq.nsmallest.

Time: O(n log k) where n = ~50 candidates and k = 5 (number of top suggestions).

Why heap over full sort? Full sort is O(n log n). Min-heap top-k is O(n log k). 
Since k ≪ n, this is much faster. For 50 candidates and k=5:
  - Full sort: 50 log 50 ≈ 282 operations
  - Min-heap: 50 log 5 ≈ 116 operations

Ranking logic: (edit_distance, -word_frequency). Smaller distance wins. Ties 
broken by frequency.

Result: Top 5 suggestions returned, ordered by quality."

================================================================================
SUMMARY (4:30 - 5:00)
================================================================================

"Let's recap:

**Stage A:** O(1) hash lookup—fast path for correct words.
**Stage B:** O(L² × 26) candidate generation, then O(m × n) distance calculation 
for each candidate. Dynamic programming avoids redundant work.
**Stage C:** O(n log k) ranking with heap.

The bottleneck is Stage B—it's 85% of runtime. Why? Character-by-character 
comparisons are the dominant operation.

The key insight: Progressive filtering. Each stage narrows the search space 
before the next, expensive stage runs. We never compare against all 554 words. 
That's how we stay fast.

Thank you."
