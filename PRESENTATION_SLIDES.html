🔍 Spell Checker

3-Stage Algorithm Pipeline

A Deep Dive into Algorithmic Design

Osman Aliyev, Łukasz Madej

🎯 The Challenge

Users type words incorrectly. How do we suggest the right correction efficiently?

Example:
Input: "recieve"
Output: "receive"

System Objective

  • ✅ Detect misspelled words instantly
  • ✅ Suggest the most likely correction
  • ✅ Handle single-letter errors, transpositions, phonetic variations
  • ✅ Scale to 500+ word dictionary with sub-millisecond response time

Why It Matters

Naive approach: Compare against all dictionary words = O(D × m × n) where D = dictionary size, m, n = word lengths.

Solution: A 3-stage pipeline that filters progressively at each stage.

📊 Input Format

  • Type: String (single word)
  • Format: Case-insensitive (normalized to lowercase)

📤 Expected Output

{
  "word": "recieve",
  "is_correct": false,
  "suggestions": ["receive", "believe", ...],
  "confidence": "high"
}

🚨 Edge Cases Handled

Edge Case Behavior Strategy
Empty input Skip all stages Early exit
Already correct Return immediately Early termination
No close matches Return empty suggestions Graceful degradation
Compound errors Caught at distance 2 Two-layer generation

Algorithm A: Hash-Based Dictionary Lookup

Role: Fast-path filter — Is the word already correct?

Data Structure: Python set (hash table)
Time Complexity: Θ(1) average case
Space Complexity: O(V) where V = vocabulary size (554 words)

Why This Complexity?

  • Hash sets use uniform hashing with O(1) amortized insertion/lookup
  • Collision resolution (chaining/probing) keeps operations constant on average
  • No sorting or traversal needed

Example Flow

  • ✅ Input "receive" → Found in hash set → Output "is_correct: true"
  • ❌ Input "recieve" → Not found → Continue to Stage B

Algorithm B: Edit Distance Candidate Generation

Role: Generate plausible corrections using Levenshtein distance

Two-Layer Strategy

Layer 1 — Distance 1 (Typo Candidates)

Generate ALL strings exactly one edit operation away:

  • Delete: each character
  • Insert: each letter a-z at each position
  • Replace: each character
  • Transpose: adjacent pairs → "recieve" → "receive" ✅

Time: O(L² × 26) where L = word length

Layer 2 — Distance 2 (Harder Errors)

Apply Layer 1 edits to each candidate. Catches errors like "recieve" → "believe"

Time: O(L⁴) but heavily pruned by dictionary membership

Algorithm B (continued): Levenshtein Distance DP

Dynamic Programming Approach

Compute actual edit distance between input and each dictionary candidate.

DP Recurrence:
If characters match: dp[i][j] = dp[i-1][j-1]
Else: dp[i][j] = 1 + min(
    dp[i-1][j],      // deletion
    dp[i][j-1],      // insertion
    dp[i-1][j-1]     // substitution
)
Time per pair: O(m × n) where m, n = word lengths
Space Optimization: Use 2 rows instead of full matrix → O(min(m,n))

Why This Complexity?

  • DP avoids recomputation via memoization
  • Without DP: exponential recomputation
  • With DP: each cell computed once
  • Two-layer filtering reduces comparisons from 554 to ~50 candidates

Algorithm C: Heap-Based Top-K Ranking

Role: Select top suggestions sorted by edit distance (primary) and frequency (tiebreaker)

Data Structure: Binary min-heap (heapq.nsmallest)
Time Complexity: O(n log k) where n = candidates, k = top results
Space Complexity: O(n)

Ranking Logic

score = (edit_distance, -word_frequency)
# Sort ascending:
# - Smaller distance wins
# - Higher frequency wins (-freq to reverse)

Why Min-Heap Over Full Sort?

Method Time When to Use
Full Sort O(n log n) Need all results sorted
Min-Heap (Ours) O(n log k) k ≪ n (5 ≪ 50)

Example: k=5 results from n=50 candidates:
Min-heap: 50 log₂ 5 ≈ 116 operations vs. Full sort: 50 log₂ 50 ≈ 282 ops → 2.4× faster

🏗️ Design Decisions

Why Hash Set Over Binary Search Tree?

Criterion Hash Set BST
Lookup Time Θ(1) avg O(log n)
Range Queries ❌ No ✅ Yes
Cache Locality Better (array) Worse (pointers)

Verdict: Hash set is optimal for membership testing. If we needed "all words starting with 'rec...'", a trie or BST would be better. But we don't.

Why Min-Heap Over Quickselect?

  • Min-Heap: O(n log k), simple, excellent cache performance
  • Quickselect: O(n) average but O(n²) worst-case, recursion overhead
  • Choice: Min-heap for stability and predictability

🔴 Primary System Bottleneck

⚠️ Edit Distance Computation (Stage B)

Profiling shows 85% of runtime spent in levenshtein_distance()

Why?

  • Each candidate pair requires O(m × n) character comparisons
  • 50 candidates × comparison threshold → ~27,700 operations
  • All other stages are negligible (Stage A: 0.1%, Stage C: 5%)

🚀 Optimization Opportunities

  1. Memoization: Cache levenshtein_distance("recieve", "receive") results
  2. Early Termination: Stop DP if distance exceeds threshold
  3. Phonetic Pre-filtering: Use Soundex/Metaphone before DP

These are deployment considerations that show how algorithmic analysis drives real optimization.

🎬 Example Run: "recieve"

Stage A: Dictionary Lookup

Is "recieve" in hash set? → No ❌
→ Proceed to Stage B

Stage B: Candidate Generation

Distance 1: ["receive"] ← Found via transpose!
Distance 2: ["believe", "perceive", ...]

Levenshtein distances:
  "recieve" vs "receive"  → 1 (one transpose)
  "recieve" vs "believe"  → 2 (replace + transpose)
  "recieve" vs "perceive" → 2 (replace + insert)

Stage C: Heap Ranking (top_k=3)

Rank 1: "receive"   (dist=1, freq=98)   ← WINNER
Rank 2: "believe"   (dist=2, freq=45)
Rank 3: "perceive"  (dist=2, freq=32)
✅ Output:
{"is_correct": false, "suggestions": ["receive", "believe", "perceive"], "confidence": "high"}

Performance: ~2ms (vs. 500ms for naive brute force) = 250× speedup

📋 Summary

Stage Algorithm Time Space Purpose
A Hash Lookup O(1) O(V) Filter correct words
B Levenshtein DP O(m×n) O(min(m,n)) Generate candidates
C Min-Heap O(n log k) O(n) Rank top-K

🎓 Key Takeaway

This project demonstrates how careful algorithmic design and appropriate data structures transform a naive O(D × m × n) problem into a fast, robust, scalable system.
